# AlphaConsole PRD

## 1. 文档信息

- 文档名称：AlphaConsole 产品需求文档
- 文档版本：v0.1
- 更新时间：2026-04-18
- 文档状态：草案

## 2. 项目概述

AlphaConsole 是一个运行在树莓派上的本地优先个人提醒与编排系统。系统通过接入 LLM API 理解用户的自然语言输入，将其转化为结构化的日程、提醒和场景化检查清单，并在合适的时间通过 TUI 界面与 ESC/POS 兼容打印机输出提醒内容。

项目的核心特点不是传统的 Todo 管理，而是“计划编排 + 场景触发 + 实体打印提醒”。其中，打印内容既包括日程计划，也包括类似飞行员起飞前检查单的 checklist，例如出门前检查雨伞、水杯、耳机、钥匙等。

后续系统需要具备可扩展性，能够逐步接入语音交互与手机端入口，但第一阶段以树莓派本地运行、可远程访问、可稳定打印为主。

## 3. 背景与问题

用户希望构建一个兼具仪式感与执行力的个人管理系统，解决以下问题：

- 自然语言记录计划后，仍需要手动整理成日历或提醒，步骤繁琐。
- 普通 Todo List 更偏长期任务管理，不适合“出门前检查”“会议前检查”“出差前检查”这类程序化清单。
- 数字提醒容易被忽略，而纸质提醒更具环境感知和存在感。
- 提醒系统如果只依赖云端或手机通知，缺少本地稳定执行能力。

## 4. 产品目标

- 支持用户用自然语言描述需求，由 LLM 将内容编排为结构化日程与提醒计划。
- 支持与用户日历集成，能够写入或同步事件信息。
- 支持在指定时间触发提醒，并将提醒内容打印到 ESC/POS 兼容打印机。
- 支持打印日程计划摘要。
- 支持打印场景化 checklist，而非仅维护普通 Todo 列表。
- 支持通过 TUI 查看、审核和管理计划、提醒、清单与打印任务。
- 支持在树莓派上稳定运行，并支持远程访问与维护。
- 为后续语音交互与手机 App 接入保留统一输入接口。

## 5. 非目标

以下内容不属于第一阶段必须完成的范围：

- 完整的通用任务管理平台
- 多人协作或家庭成员共享系统
- 公网直接暴露的开放服务
- 原生 iOS / Android App
- 完整自动化的无确认 AI 决策系统

## 6. 目标用户

当前目标用户为单一主用户，即设备拥有者本人。

该用户的典型特征：

- 习惯用自然语言描述计划和提醒需求
- 希望提醒系统既智能又可控
- 需要可复用的场景化检查流程
- 愿意在桌面、玄关、工作区等场景使用实体打印提醒

## 7. 核心使用场景

### 7.1 日程编排

用户输入一句自然语言，例如：

“明天下午三点和客户开会，提前一小时提醒我准备资料。”

系统需要：

- 理解事件内容与时间
- 生成结构化日程
- 为该日程生成提醒时间
- 视配置写入日历
- 在提醒时间到达时打印提醒单

### 7.2 今日计划打印

用户希望系统在每天固定时间打印一张“今日计划”，内容包含：

- 当天的重要事件
- 对应时间段
- 简短备注
- 必要提醒项

### 7.3 出门前检查清单

用户输入一句自然语言，例如：

“明天早上出门前提醒我检查雨伞、水杯、耳机和钥匙。”

系统需要：

- 识别这是 checklist 场景，而非普通 todo
- 生成一个带触发时间的检查清单实例
- 在指定时间打印一张可勾选的小票式清单

### 7.4 场景模板复用

用户未来可能定义固定模板，例如：

- 上班出门检查
- 出差出发检查
- 开会前检查
- 直播前设备检查

系统需要支持复用模板，并按场景生成新的 checklist run。

### 7.5 远程访问

用户不一定总在树莓派旁边，因此系统需要支持远程访问，用于：

- 查看当前计划和提醒状态
- 新增自然语言输入
- 查看打印记录与错误状态
- 进行基础维护

### 7.6 后续语音与手机扩展

未来用户希望：

- 通过语音直接说出安排需求
- 通过手机 App 或手机网页发起输入

因此系统需要将“输入方式”与“编排执行核心”解耦。

## 8. 产品原则

- 本地优先：核心调度、提醒与打印不依赖持续云连接才能工作。
- LLM 负责理解与编排，不直接成为系统真相源。
- 所有关键计划均应落地为结构化本地数据。
- checklist 是“程序化检查流程”，不是普通 todo。
- 打印内容应简洁、明确、可执行。
- 系统应支持逐步扩展，而不是一次性做成全能助手。

## 9. 功能需求

### 9.1 自然语言输入

系统需要支持用户输入自然语言命令，用于表达：

- 新建日程
- 调整提醒
- 创建 checklist
- 基于特定事件生成 checklist
- 查询某天计划

第一阶段至少支持：

- TUI 内输入
- 远程访问条件下的命令输入

后续支持：

- 语音输入
- 手机端输入

### 9.2 LLM 编排

系统需要调用 LLM API，将自然语言转换为结构化结果。结构化结果至少需要覆盖：

- 意图类型
- 事件标题
- 事件时间
- 提醒时间
- checklist 标题
- checklist 项目
- 备注
- 是否需要用户确认

系统需要支持以下意图分类：

- `event`
- `checklist`
- `event + checklist`
- `clarification_required`

### 9.3 结构化校验

在执行前，系统需要对 LLM 输出进行本地校验，避免错误写入或错误提醒。校验内容至少包括：

- 时间格式是否合法
- 时区是否明确
- 触发时间是否早于当前时间
- checklist 项目是否为空
- 关键字段是否缺失
- 是否存在明显冲突或歧义

必要时系统应要求用户确认。

### 9.4 日历接入

系统需要支持接入用户日历，用于：

- 创建事件
- 更新事件
- 查询某日计划
- 将本地计划与外部日历关联

第一阶段支持一个日历源即可。日历写入应与本地记录建立唯一映射关系，避免重复创建事件。

### 9.5 提醒调度

系统需要具备本地调度能力，支持：

- 单次提醒
- 相对事件时间的提醒
- 固定时间打印今日计划
- 到时触发打印
- 树莓派重启后恢复未完成任务

### 9.6 Checklist 模块

系统需要支持两类 checklist：

- 模板型 checklist
- 某次触发生成的 checklist run

每个 checklist 至少包含：

- 标题
- 场景
- 条目列表
- 触发时间
- 状态
- 来源

条目需要支持以下类型：

- 常规项
- 关键项
- 条件项

### 9.7 打印模块

系统需要支持通过 ESC/POS 兼容打印机打印以下内容：

- 单条提醒
- 今日计划摘要
- checklist
- 错误或异常提示单

打印内容应满足：

- 适合热敏小票宽度
- 文本层次清晰
- 支持中英文混排或具备图像兜底方案
- checklist 可打印为可勾选格式

### 9.8 TUI 管理界面

系统需要提供基于字符界面的 TUI，用于：

- 新增自然语言输入
- 查看今日计划
- 查看待触发提醒
- 查看 checklist 模板和实例
- 查看打印状态
- 审核待确认的 LLM 结果
- 手动补打或重打

### 9.9 远程访问

系统需要支持远程访问树莓派上的服务，以便用户在外部网络环境下使用。远程访问至少需要满足：

- 安全接入
- 不直接暴露不必要公网端口
- 可远程维护
- 可远程提交输入或查看状态

### 9.10 日志与可观测性

系统需要记录关键日志，至少包括：

- 用户输入
- LLM 输出
- 校验结果
- 日历写入结果
- 调度触发记录
- 打印成功或失败记录

## 10. 数据对象

系统第一阶段建议具备以下核心数据对象：

### 10.1 Event

表示日历事件或计划项，字段至少包括：

- 本地唯一 ID
- 标题
- 开始时间
- 结束时间
- 时区
- 备注
- 外部日历 ID
- 创建来源

### 10.2 ReminderJob

表示一个待执行提醒任务，字段至少包括：

- 本地唯一 ID
- 关联对象类型
- 关联对象 ID
- 触发时间
- 触发动作
- 当前状态

### 10.3 ChecklistTemplate

表示可复用的场景清单模板，字段至少包括：

- 本地唯一 ID
- 模板名称
- 场景标签
- 条目列表
- 默认触发偏移

### 10.4 ChecklistRun

表示某次实际触发的检查单实例，字段至少包括：

- 本地唯一 ID
- 模板来源
- 标题
- 条目快照
- 触发时间
- 关联事件
- 打印状态

### 10.5 PrintJob

表示一次打印任务，字段至少包括：

- 本地唯一 ID
- 打印内容类型
- 内容快照
- 目标打印机
- 提交时间
- 执行状态
- 错误信息

## 11. 典型数据流

### 11.1 日程提醒

用户输入自然语言 -> LLM 输出结构化事件与提醒 -> 本地校验 -> 写入本地数据库 -> 视配置写入外部日历 -> 创建 ReminderJob -> 到时触发 -> 生成打印内容 -> 打印机输出

### 11.2 Checklist 提醒

用户输入自然语言 -> LLM 输出 checklist 或 event + checklist -> 本地校验 -> 创建 ChecklistRun -> 创建 ReminderJob -> 到时触发 -> 渲染 checklist 小票 -> 打印机输出

### 11.3 模板复用

用户选择或触发某个 ChecklistTemplate -> 系统生成 ChecklistRun -> 写入本地数据库 -> 创建 ReminderJob -> 到时打印

## 12. 非功能需求

### 12.1 可靠性

- 树莓派重启后，待触发提醒不能丢失
- 打印失败时需要记录状态并支持重试
- 外部日历或 LLM API 短暂不可用时，系统不能导致本地数据损坏

### 12.2 安全性

- 远程访问应采用安全网络接入方式
- API 密钥和用户凭证需要安全存储
- 系统应尽量避免开放公网写接口

### 12.3 可扩展性

- 输入通道需要与核心逻辑解耦
- 打印渲染需要与 TUI 渲染解耦
- 未来应可增加语音和手机端而不重写核心逻辑

### 12.4 可维护性

- 模块边界清晰
- 日志明确
- 配置可管理
- 便于替换打印机、日历源或 LLM 服务商

## 13. 第一阶段 MVP 范围

第一阶段建议实现以下最小可用能力：

- 在树莓派本地运行
- 接入一台 ESC/POS 兼容打印机
- 提供一个 TUI 输入与查看界面
- 接入一个 LLM API
- 支持自然语言创建单次日程提醒
- 支持自然语言创建单次 checklist 提醒
- 支持打印提醒单和 checklist 小票
- 支持本地持久化与调度恢复
- 支持基础远程访问

第一阶段可暂缓：

- 复杂多轮对话
- 高级规则引擎
- 天气联动等条件触发
- 原生手机 App
- 完整语音交互闭环

## 14. 第二阶段方向

- 支持 checklist 模板库
- 支持每日自动打印今日计划
- 支持与外部日历双向关联
- 支持待确认队列与更细粒度审核
- 支持手机网页入口
- 支持语音输入

## 15. 第三阶段方向

- 支持条件型 checklist，例如根据天气、地点、事件类型动态增减条目
- 支持多提醒渠道联动，例如打印 + 手机通知
- 支持更复杂的计划编排与冲突处理
- 支持长期行为模式与个人习惯学习

## 16. 成功标准

若满足以下条件，可认为第一阶段达到成功：

- 用户能用一句自然语言创建一条有效提醒
- 用户能用一句自然语言创建一张可打印 checklist
- 系统能在设定时间自动打印相应内容
- 树莓派重启后提醒任务仍能恢复
- 用户能远程访问系统查看状态并新增输入

## 17. 当前待确认问题

以下问题尚未完全确定，后续需要在技术设计阶段确认：

- 第一阶段接入哪一种日历服务
- 中文打印采用原生字符编码还是位图兜底
- 远程访问入口仅限 SSH/TUI，还是同时提供轻量 Web 入口
- checklist 是否需要支持手动回填“已完成”状态
- 打印机是否需要双模式输出，例如即时提醒与定时摘要

## 18. 总结

AlphaConsole 的本质不是一个普通待办工具，而是一个围绕“自然语言编排、时间触发、场景检查、实体打印提醒”构建的个人执行系统。第一阶段应聚焦于本地可靠运行、结构化编排、清晰打印和基础远程访问，在此基础上逐步扩展到模板化 checklist、手机入口和语音交互。
